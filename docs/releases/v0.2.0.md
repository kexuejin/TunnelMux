# TunnelMux v0.2.0

TunnelMux `v0.2.0` turns the GUI into the primary local workflow instead of a thin wrapper around daemon APIs.

If your local stack now looks like “frontend + API + docs + callbacks”, this release makes that workflow much easier to start, manage, and debug without stitching together `cloudflared`, `ngrok`, and route scripts by hand.

## Highlights

- GUI-first local tunnel workflow
  - current tunnel stays in focus
  - public URL, services, and core actions stay on one screen
  - tunnel creation, editing, and deletion are built into the app
- Real multi-tunnel runtime
  - multiple tunnel profiles can coexist
  - runtime state, routes, diagnostics, logs, metrics, and upstream health are now tunnel-scoped
  - deleting a tunnel cleans up daemon-owned runtime and routes
- Better provider handling
  - `cloudflared` and `ngrok` are handled as first-class providers
  - provider-specific actions and status are surfaced more clearly in the GUI
- Lower-noise troubleshooting
  - status messaging is quieter by default
  - diagnostics stay available on demand instead of dominating the main UI
  - error details are shown only when they are actually useful
- Better distribution
  - GUI installers ship alongside raw archives
  - README is now GUI-first and bilingual
  - macOS Gatekeeper workarounds are documented for unsigned builds

## Notes

- Current macOS GUI builds may still be unsigned, so first launch can trigger “App is damaged” or “cannot verify developer” prompts.
- For trusted downloads, use Finder `Open`, then `System Settings` → `Privacy & Security` → `Open Anyway` if needed.
- If you validate release artifacts locally, build the release binaries before running `cargo tauri build`, otherwise `target/release/tunnelmuxd` and `target/release/tunnelmux-cli` can remain stale.
- Release handoff should also run the `GUI Easy-Path Smoke Check` in `docs/RELEASING.md` to verify missing-provider guidance, install-command copy, quick-tunnel start, service add, and public URL sharing.
